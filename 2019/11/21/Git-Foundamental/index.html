Git Foundamental

Posted on 2019-11-21 | In notes

These steps are done under the local environment, which means I Won’t push any data to remote repository.

git init OR clone

  1. Create a new folder: mkidr 'folder name';
  2. Enter this new folder: cd 'folder name';
  3. Initialize repository: git init, terminal would output some words that include ‘master’. This method will create a perfectly new repository whitout any project data but configuration files;

    OR we can:
  4. Get an identical repository: git clone url [new folder name]. After this way we get a totally same project as remote repository that located in url. If we add a new folder name as parameter, this project will be downloaded to it, or datas will be in the peoject name folder;

Add commits to repository

  1. A project controlled by git, there will is conception that about three main part: Repository, Staging index and Working directory. All just modifyed and created resources are located in working directory. After excuted git add those resources will be staged as they are moved to staging index. Let’s make a demostration as below:

  2. git commit -m "some information about this commit", this command moves files from staging index to repository. If there is no -m and additional information, git will prompt a editor that let users to add some additional information.

  3. git diff is used to see changes that have been made but haven’t been committed, yet.
  4. git log is used to display all of the commits of a repository. Like this:

git log --oneline --decorate --graph --all


--graph flag adds the bullets and lines to the leftmost part of the output.

Tagging and branching

  1. If we want to add a marker on a specific commit, the git tag -a tagName [sha] can help us. The tag does not move around as new commits are added. Along with sha, the is tagged to a certain commit.
  2. git tag -d v1.0 deletes tag ‘v1.0’.

  3. git branch shows the branches and reveals the current branch.

  4. git checkout sidebar: we created the new sidebar, no new commits will be added to it since we haven’t switched to it, yet. If we made a commit right now, that commit would be added to the master branch, not the sidebar branch. So to switch between branches, we need to use Git’s checkout command.If you provide the -b flag, you can create a branch and switch to it all in one command.

Merging

  1. To combine Git branches: git merge <name-of-branch-to-merge-in>;

  2. merge conflict: If a merge conflict does occur, Git will try to combine as much as it can, but then it will leave special markers (e.g.

There are special characters called “Ancestry References” that we can use to tell Git about these relative references. Those characters are:

^ – indicates the parent commit
~ – indicates the first parent commit

Here’s how we can refer to previous commits:

The main difference between the ^ and the ~ is when a commit is created from a merge. A merge commit has two parents. With a merge commit, the ^ reference is used to indicate the first parent of the commit while ^2 indicates the second parent. The first parent is the branch you were on when you ran git merge while the second parent is the branch that was merged in.

It’s easier if we look at an example. This what my git log currently shows:

Since HEAD points to the 9ec05ca commit:
HEAD^ is the db7e87a commit
HEAD~1 is also the db7e87a commit
HEAD^^ is the 796ddb0 commit
HEAD~2 is also the 796ddb0 commit
HEAD^^^ is the 0c5975a commit
HEAD~3 is also the 0c5975a commit
HEAD^^^2 is the 4c9749e commit (this is the grandparent’s (HEAD^^) second parent (^2))

  1. Git Reset’s Flags are --mixed,--soft,--hard

    git reset --mixed HEAD^ will take the changes made in commit and move them to the working directory.
    git reset --soft HEAD^ will take the changes made in commit and move them directly to the Staging Index.
    git reset --hard HEAD^ will take the changes made in commit and erases them.
